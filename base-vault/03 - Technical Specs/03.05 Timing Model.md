# Timing Model

## O Problema do Timing

Firmware antigo frequentemente tem polling loops com timeouts hardcoded que assumem latência específica do ASIC original. Se o novo hardware for:

- **Muito mais rápido**: polling lê "pronto" antes do periférico estar realmente pronto
- **Mais lento**: timeout estoura antes da operação completar

## Timing Compensation

```rust
struct TimingCompensation {
    forced_delays: Vec<ForcedDelay>,
    warnings: Vec<TimingWarning>,
}

struct ForcedDelay {
    operation: Operation,
    min_delay: Duration,
    reason: String,
}

struct TimingWarning {
    operation: Operation,
    original_latency: Duration,
    target_latency: Duration,
    risk: String,
}

fn calculate_compensation(original: &TimingProfile, target: &TimingProfile) -> TimingCompensation {
    let mut comp = TimingCompensation::default();
    
    for (op, orig_lat, target_lat) in zip_operations(original, target) {
        if target_lat < orig_lat * 0.5 {
            comp.forced_delays.push(ForcedDelay {
                operation: op,
                min_delay: orig_lat * 0.8,
                reason: "Polling timeout prevention",
            });
        }
        if target_lat > orig_lat * 2.0 {
            comp.warnings.push(TimingWarning {
                operation: op,
                original_latency: orig_lat,
                target_latency: target_lat,
                risk: "May cause application timeout",
            });
        }
    }
    comp
}
```

## Classe de Latência por Tipo de Bloco

| Tipo de Bloco | Latência Típica | Tolerância |
|--------------|----------------|------------|
| GPU Frame | 4-16ms | ±20% |
| DMA Setup | 0.5-5µs | ±50% |
| Audio DMA | 2-10ms (buffer) | ±10% |
| I2C Transfer | 0.1-10ms (dep. do clock) | ±100% |
| SPI Transfer | 0.01-1ms | ±50% |
| Interrupt Response | 0.5-5µs | ±100% |
| Memory Access | 10-100ns | ±20% |
| USB Transaction | 0.1-10ms | ±50% |
