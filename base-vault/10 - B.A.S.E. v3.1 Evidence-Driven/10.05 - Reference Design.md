# Reference Design

> *Saída principal do pipeline. Um YAML descritivo — não uma PCB final.*

## Formato

```yaml
# reference_design.yaml
design:
  title: "LK Bootloader Reconstruction"
  version: 1
  source: "lk-sign.bin (disassembly)"

  architecture:
    cpu:
      part: RP2350A
      package: QFN-56
      price: 1.50

    memory:
      - type: qspi_flash
        part: W25Q128JV
        size: 16MB
        interface: spi
        price: 0.80

    audio:
      part: PCM5102A
      interface: i2s
      price: 1.50

  contracts:
    total: 5
    satisfied: 5
    violations: []

  bom:
    total_parts: 3
    estimated_cost: 3.80
    currency: USD

  pcb:
    type: engineering_draft
    layers: 2
    notes:
      - "Engineering draft — requires layout review"
      - "Power tree not included"
      - "Decoupling capacitors TBD"

  validation:
    status: pending
    contracts_verified: false
```

## O que o Contract Solver GARANTE

✅ **Contrato satisfeito** — o componente atende aos requisitos

O que ele NÃO garante:
❌ A PCB vai funcionar (depende de layout, alimentação, ruído, impedância)
❌ O firmware vai bootar (depende de implementação)
❌ A integridade elétrica (depende de layout)

## Path to Real R3 (wedge UART)

Heurísticas honestas do mapper / design:

1. **MCU ≻ FPGA** para Uart/Spi/I2c/Timer — candidatos FPGA só se não houver MCU na DB.
2. Empate de score → preferência de categoria, depois **menor `price_1k`**.
3. Pinout omitido no YAML: MCU/CPU assumem **32 GPIO** nos contratos Count; FPGA não.
4. Gate de validação: `contracts_verified` se **≥70%** satisfeitos; status `contracts_satisfied` se zero violations.
5. Budget (`synth --max-bom-cost`): partes sem preço ou `price_1k: 0` **não** entram.

Piloto (`examples/pilot/run.sh`): tipicamente **RP2040** @ $0.70, interface `uart`, **3/3** contratos.

Testes: `cargo test -p base-core --test pilot_design`
