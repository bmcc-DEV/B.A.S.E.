## PRINCIPAIS GARGALOS

### 1. Inferência comportamental frágil

**Problema:** O `hardware_spec.yaml` gerado classifica dispositivos com pouca evidência.

```yaml
kind: InterruptController
confidence: 0.125
```

e:

```yaml
kind: GPU
```

para um bloco chamado `Doorbell_4`.

**Causa:** O analisador atual usa heurísticas simples (ratio de writes/reads) sem validação semântica.

**Impacto:** O restante do pipeline (synth → pcb → fw) opera sobre classificações de baixa confiança.

### 2. Falta Event Graph

**Problema:** O grafo gerado é **estrutural** (quem conversa com quem), não **causal** (quem faz o quê).

```
Hoje:     CPU → MMIO → GPU
Preciso:  WRITE → DMA_START → DMA_COMPLETE → IRQ
```

**O que falta:**
- Eventos (write, read, DMA, IRQ) como nós de primeira classe
- Relações causais entre eventos
- Timing (latências entre eventos)
- Estados (idle → active → done)

### 3. Firmware não conhece o comportamento

**Problema:** A HAL gerada traduz endereços, mas não modela comportamento.

```c
{ 0x00004000, 0x0aac, 0x00104000, 1 }  // só traduz endereço
```

**O que deveria fazer:**

```c
if (addr == GPU_CONTROL && value == 1) {
    gpu_start_dma();           // comportamento!
    timing_delay(150, 400);    // respeita contrato temporal!
}
```

### 4. BOM é placeholder

**Problema:** O synthesizer atualmente faz `Block → Componente` sem análise de requisitos.

```
Hoje:     Bloco → Componente (match heurístico)
Preciso:  Requisitos → Constraint Solver → Arquitetura → Componentes
```

## PRIORIDADES DE MELHORIA

### Prioridade 1 — BIR como modelo central

Criar o Behavioral IR (BIR) como o modelo semântico central do pipeline:

```text
Firmware → Analyze → BIR → HardwareSpec → Synth → PCB + FW
```

O BIR já está implementado em `base-bir/` mas não é o centro do pipeline — o `HardwareSpec` ainda é. A migração para BIR como formato canônico é a mudança arquitetural mais importante.

### Prioridade 2 — Confidence em tudo

Cada inferência precisa expor sua confiança:

```yaml
gpu:
  confidence: 0.34
dma:
  confidence: 0.91
```

Isso já existe parcialmente (`FunctionalBlock.confidence`) mas não é propagado corretamente para o YAML de saída.

### Prioridade 3 — Behavior Graph → Event + Timing Graph

O DOT export agora gera **dois grafos**:

- `behavior_graph.dot` — estrutural (quem conversa com quem)
- `event_graph.dot` — causal + temporal (eventos, IRQs, DMA, timing)

Ver [[09 - B.A.S.E. v2 Expansion/09.03 - Knowledge Graph]].

### Prioridade 4 — Component Genome Database

O `ComponentDb` tem 51 entradas, mas o formato precisa ser enriquecido com:
- Timing (latências por operação)
- Compatibilidade (componentes equivalentes)
- Preço e disponibilidade atuais

### Prioridade 5 — Constraint Solver (Z3 + ILP)

Em vez de match heurístico, usar solver para encontrar a arquitetura ótima:

```yaml
constraints:
  cost_max: 20
  power_max: 5W
  layers_max: 4

solution:
  cpu: rp2350
  ram: sram
  audio: tlv320
```
